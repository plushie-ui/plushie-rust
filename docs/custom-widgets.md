# Custom Widgets

Plushie supports two ways to build reusable widgets: composite
widgets and native widgets. This document covers native widgets:
renderer-side primitives that implement iced-based drawing and are
compiled into a custom renderer binary by `cargo plushie build`.

For the other style, see [When to write a native widget](#when-to-write-a-native-widget)
below; it contrasts native widgets against composite widgets and
explains when to reach for each.

## When to write a native widget

Plushie offers two widget paths with different cost profiles and
different reach.

**Composite widgets** (the `plushie::widget::Widget` trait) live in
the app crate. They are iced-free, depend only on `plushie` and
`plushie-core`, and compose existing widgets through `View`.
Composite widgets work in any mode (direct, wire, WASM) without a
custom renderer build. This is the right choice for application
shapes (rating stars, form sections, list items) that combine
built-in widgets into a higher-level unit. See the
[Core Widget Guide](core-widget-guide.md) for composite widget
patterns.

**Native widgets** are Rust crates that implement the
`PlushieWidget` trait from `plushie-widget-sdk`. They compile into
the renderer binary, so wire-mode apps need `cargo plushie build` to
produce a widget-aware renderer. Native widgets are the right choice
when the widget needs renderer primitives beyond what the built-in
set provides: custom drawing, direct access to iced's widget
library, stateful iced widgets, or protocol-level event shapes that
composites cannot express.

The rest of this document is about native widgets.

## Widget crate architecture

Native widget crates use a single-crate layout with an `impl` feature
flag:

- **Default features (stub-only).** The crate depends on
  `plushie-core` (iced-free) and exposes the widget's typed struct
  plus builder methods. App crates pull this form. The generated
  tree nodes are plain `TreeNode` values carrying the widget's
  `type_name` and props.
- **`impl` feature.** Adds the `PlushieWidget` trait implementation.
  Depends on `plushie-widget-sdk`, which pulls iced. The custom
  renderer generated by `cargo plushie build` enables this feature
  so the iced-based drawing code is compiled in.

This shape keeps app crates iced-free. An app in wire mode never
touches iced directly: the widget stub is small and compiles
quickly, and the iced-heavy implementation stays inside the
renderer binary.

### Crate skeleton

```
my-gauge/
  Cargo.toml
  src/
    lib.rs
```

**`Cargo.toml`**

```toml
[package]
name = "my-gauge"
version = "0.1.0"
edition = "2024"

[package.metadata.plushie.widget]
type_name = "my_gauge"
constructor = "my_gauge::factory::MyGaugeFactory::new()"

[features]
impl = ["dep:plushie-widget-sdk"]

[dependencies]
plushie-core = "0.6"
plushie-core-macros = "0.6"
plushie-widget-sdk = { version = "0.6", optional = true }
```

The `[package.metadata.plushie.widget]` table is what `cargo plushie
build` scans for during widget discovery. Both `type_name` and
`constructor` are required. See the
[Metadata reference](build-tool.md#metadata-reference) in the build
tool documentation for full field semantics.

## The `widget!` macro

`plushie_core::widget!` is a function-like macro that declares a
widget struct, generates its builder methods, and produces a
metadata constant the build tool consumes.

It is re-exported from `plushie_widget_sdk::prelude` too, so a
widget crate only needs one `use` for both the macro and the
implementation prelude.

### Minimal example

```rust
use plushie_core::widget;

widget! {
    /// Circular gauge.
    #[widget(type_name = "my_gauge")]
    pub struct MyGauge {
        /// Current value, clamped to `[0.0, max]` at render time.
        pub value: f32,
        /// Full-scale value.
        pub max: f32,
    }
}
```

### Attributes

The `#[widget(...)]` attribute on the struct accepts:

- `type_name = "..."` (required): the wire-protocol type name. Must
  match the `type_name` in `Cargo.toml`.
- `crate = "..."` (optional): source crate name embedded in the
  metadata JSON for tooling.

The factory constructor string lives in
`[package.metadata.plushie.widget]` in the widget crate's
`Cargo.toml`. Keeping one copy means the scaffold does not drift and
authors cannot edit one and forget the other. Passing `constructor`
to the macro is a hard error; see the
[Build tool metadata reference](build-tool.md#metadata-reference).

### Generated items

For the example above, the macro emits:

- `pub struct MyGauge { pub id: String, pub value: Option<f32>, pub max: Option<f32> }`
- `MyGauge::new(id)` constructor.
- `MyGauge::type_name()` associated fn returning `"my_gauge"`.
- Fluent setters: `MyGauge::value(f32)`, `MyGauge::max(f32)`.
- `impl From<MyGauge> for plushie_core::protocol::TreeNode`.
- `pub const PLUSHIE_WIDGET_METADATA: &str` with a JSON snippet the
  build tool reads.

Fields are stored as `Option<T>` so the builder tracks which setters
were called. The `From<Widget> for TreeNode` conversion only encodes
`Some(_)` fields into the node's props.

### Optional events block

A trailing `events { ... }` block generates a sibling enum with
`WidgetEvent` derived, giving the widget a typed event surface:

```rust
use plushie_core::widget;

widget! {
    #[widget(type_name = "my_gauge")]
    pub struct MyGauge {
        pub value: f32,
        pub max: f32,
    }

    events {
        ValueChanged(f32),
    }
}
```

The block expands to `pub enum MyGaugeEvent { ValueChanged(f32) }`
with the derive attached.

## Constructor and factory pattern

Widgets have two entry points that serve different audiences:

- **`MyGauge::new(id)` on the widget struct.** App code calls this
  in `App::view` to produce a `View`. Each call is a fresh builder
  carrying that specific widget instance's ID and props.
- **Factory type with a zero-arg `new()`.** The custom renderer
  calls this once per session to register a renderer-side instance
  that handles every tree node with the matching `type_name`. The
  `constructor` metadata names this factory, not the widget struct.

By convention the factory lives in an `impl`-feature-gated module
and follows the name `<Widget>Factory`. For `MyGauge`, the factory
is `my_gauge::factory::MyGaugeFactory`.

```rust
#[cfg(feature = "impl")]
pub mod factory {
    use plushie_widget_sdk::prelude::*;

    #[derive(PlushieWidget, Default)]
    #[plushie_widget(type_name = "my_gauge")]
    pub struct MyGaugeFactory;

    impl MyGaugeFactory {
        #[must_use]
        pub const fn new() -> Self {
            Self
        }
    }

    impl PlushieWidgetRender for MyGaugeFactory {
        fn render<'a>(
            &'a self,
            node: &'a TreeNode,
            ctx: &RenderCtx<'a>,
        ) -> PlushieElement<'a> {
            let value = node.prop_f32("value").unwrap_or(0.0);
            let max = node.prop_f32("max").unwrap_or(1.0);
            let ratio = (value / max).clamp(0.0, 1.0);
            container(text(format!("{:.0}%", ratio * 100.0)))
                .padding(8)
                .into()
        }
    }
}
```

`#[derive(PlushieWidget)]` with `type_name` generates the
`type_names()` and `fresh_for_session()` methods on the
`PlushieWidget` trait. `PlushieWidgetRender` holds just the `render`
body. `PlushieElement<'a>` is shorthand for
`iced::Element<'a, Message, iced::Theme, iced::Renderer>`. Use
`PlushieElement<'a, R>` only when writing renderer-generic code.

## Implementing `PlushieWidget`

`PlushieWidget` is the trait the renderer calls to turn a widget's
tree node into an iced `Element`. The full trait lives in
`plushie_widget_sdk::registry`; most widgets only implement two
methods directly and get the rest from the derive macro.

Required methods:

- `type_names(&self) -> &[&str]`: the widget type names this
  implementation handles. Usually `&["my_gauge"]` for a single-type
  widget.
- `render(&'a self, node: &'a TreeNode, ctx: &RenderCtx<'a>) -> PlushieElement<'a>`:
  produces an iced `Element` from the node's props and children.
- `fresh_for_session(&self) -> Box<dyn PlushieWidget>`: returns
  a fresh instance for each session when the renderer runs with
  `--max-sessions > 1`. Isolates per-session state.

Optional lifecycle hooks (see the trait docs for full signatures):

- `prepare(&mut self, node, window_id, theme)`: populate mutable
  state from the tree node during `apply()`. Read it back
  immutably in `render()`. Used by stateful widgets (text editor,
  markdown) that need to own iced state across frames without
  interior mutability.
- `handle_message(&mut self, msg)`: process widget-specific iced
  messages (for example, drag updates from a custom interaction
  loop).
- `handle_widget_op(&mut self, op)`: handle wire-protocol widget
  ops (`focus`, `scroll_to`, etc.).
- `cleanup_stale(&mut self, live_ids)`: prune per-instance state
  after each tree walk. Receives the set of live
  `(window_id, node_id)` keys; drop anything not in the set.
- `infer_a11y(&self, node)`: auto-infer accessibility properties
  from the node's props.
- `event_specs(&self) -> &[EventSpec]`: declare the event families
  the widget can emit.
- `command_specs(&self) -> &[CommandSpec]`: declare the commands the
  widget accepts.

When the derive macro is not a good fit (multi-type widgets, custom
`type_names` logic, optional methods that need different bodies per
widget), implement `PlushieWidget` directly:

```rust
use plushie_widget_sdk::prelude::*;

pub struct MyGaugeFactory;

impl PlushieWidget for MyGaugeFactory {
    fn type_names(&self) -> &[&str] {
        &["my_gauge"]
    }

    fn render<'a>(
        &'a self,
        node: &'a TreeNode,
        ctx: &RenderCtx<'a>,
    ) -> PlushieElement<'a> {
        // ...
        todo!()
    }

    fn fresh_for_session(&self) -> Box<dyn PlushieWidget> {
        Box::new(Self)
    }
}
```

See the `plushie-widget-sdk` rustdoc for the full trait surface and
the per-method contracts.

## Testing widgets

### `WidgetTestSession` for composite widgets

Composite widgets implementing `plushie::widget::Widget` use
`plushie::test::WidgetTestSession` for headless MVU testing. The
harness hosts the widget inside a `window > column` container and
records every event the widget emits:

```rust
use plushie::test::WidgetTestSession;

let mut session = WidgetTestSession::<StarRating>::start("stars");
session.click("star_3");
let (family, value) = session.last_event().unwrap();
assert_eq!(family, "select");
```

`start_with_props` sets initial props, and `session()` /
`session_mut()` expose the underlying `TestSession` for the full
interaction API (assertions, a11y checks, keyboard presses).

### `TestEnv` for native widgets

Native widgets use `plushie_widget_sdk::testing::TestEnv`, which
owns a `RenderCtx` and lets the test drive the widget's render
cycle directly:

```rust
use plushie_widget_sdk::prelude::*;
use plushie_widget_sdk::testing::*;

#[test]
fn gauge_renders_value() {
    let test = TestEnv::default();
    let mut widget = MyGaugeFactory::new();
    let node = node_with_props("g1", "my_gauge", serde_json::json!({
        "value": 0.5,
        "max": 1.0,
    }));
    let _element = test.prepare_and_render(&mut widget, &node, "main");
    // Assert on shared state, on generated events, or on
    // widget-local cache bookkeeping as appropriate.
}
```

The module also exposes `node()`, `node_with_props()`,
`node_with_children()`, and `node_with_props_and_children()` so
tests can construct tree inputs without pulling in the full
protocol crate.

For event bridging, `TestEnv::handle_message_events` drives
`handle_message` and returns the emitted `OutgoingEvent` values
directly, flattening `Handled(v)` and `Fallthrough` into a
uniform vector.

## Publishing

Widget crates publish to crates.io like any other Rust library.
Key points:

- **Cargo.toml metadata.** Include `[package.metadata.plushie.widget]`
  as shown above. Missing or malformed metadata means
  `cargo plushie build` will not discover the widget.
- **Feature flags.** Keep the `impl` feature optional so the crate
  compiles in iced-free contexts.
- **Version alignment.** Match the minor version of `plushie-core`
  and `plushie-widget-sdk` that the crate depends on. The renderer's
  trait surface is gated on a compatible minor version; using a
  newer widget SDK than the renderer expects will fail at compile
  time inside the generated workspace.

Before publishing, delete any development aids that reference a
local plushie-rust checkout: the scaffolder emits a `.cargo/config.toml`
with a `paths = [...]` override when `PLUSHIE_RUST_SOURCE_PATH` is set,
and that file should not ship to crates.io.

## Scaffolding

`cargo plushie new-widget <name>` scaffolds a widget crate in one
shot. The scaffolder:

- Validates the name (ASCII kebab-case, no leading or trailing
  dashes, not a shadow of a built-in type name).
- Creates the target directory (default `./native/<name>`).
- Writes `Cargo.toml` with the `[package.metadata.plushie.widget]`
  table and an `impl` feature flag.
- Writes `src/lib.rs` with a `widget!` invocation, a factory type,
  and a `todo!()` placeholder in the `render` body.
- When `PLUSHIE_RUST_SOURCE_PATH` is set, rewrites dependencies to point
  at the local plushie-rust checkout and emits a
  `.cargo/config.toml` forwarding the `plushie-iced` override. The
  same mechanism the workspace uses when developing plushie-rust
  crates against the vendored iced fork.

```sh
cargo plushie new-widget my-gauge
cargo plushie new-widget my-gauge --path crates/my-gauge
```

Customize the scaffolded crate by:

- Replacing the stub fields in the `widget!` macro with the real
  props.
- Filling in the `render` body in the `factory` module.
- Adding an `events { ... }` block if the widget emits typed events.
- Adding prop-validation helpers or a11y inference if needed.

## Integration with an app

An app picks up a widget by adding it as a regular dependency:

```toml
[dependencies]
plushie = { version = "0.6", features = ["wire"] }
my-gauge = { version = "0.1", path = "native/my-gauge" }
```

No widget registration in the app's own code is needed. The app uses
the widget's builder:

```rust
use plushie::prelude::*;
use my_gauge::MyGauge;

fn view(model: &Model, _widgets: &mut WidgetRegistrar) -> ViewList {
    window("main")
        .child(MyGauge::new("battery").value(model.battery).max(1.0))
        .into()
}
```

When `cargo plushie build` runs, it walks the dep graph, finds
`my-gauge` via its `[package.metadata.plushie.widget]` table, and
wires it into the generated renderer workspace with the `impl`
feature enabled. The resulting renderer binary handles the
`my_gauge` tree node type.

From that point on, [wire-mode binary discovery](build-tool.md#binary-discovery)
picks up the custom binary ahead of any downloaded stock renderer,
so `cargo run` just works.

See [the build tool reference](build-tool.md) for details on the
workspace generator, binary discovery, mode precedence, and the
`cargo plushie doctor` diagnostic.
